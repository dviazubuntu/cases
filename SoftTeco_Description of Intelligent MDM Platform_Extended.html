Technical Architecture

Section 9: Kubernetes & GPU Management Architecture

The platform co-schedules CPU-bound application workloads and GPU-intensive AI inference jobs. The orchestration substrate scales with the deployment tier — from serverless containers for the MVP to a full GPU-scheduling cluster for production and enterprise.

Deployment Mode Selection

Mode Platform When to use
MVP / Pilot Azure Container Apps (serverless Kubernetes) No cluster management needed; scales to zero; supports GPU; 2–3 week timeline
Production (Cloud) Azure Kubernetes Service (AKS) private cluster Full control, advanced GPU scheduling (MIG, time-slicing), 4–6 week timeline
On-Premises Rancher RKE2 or kubeadm Air-gap requirements, existing DC investment

🟢 MVP / Pilot GPU Profile — CPU-Only Viable

  • GPU: 0–1× T4 (optional) — CPU-only viable for MVP with quantized LLMs.
  • GPU can be added later; the platform gracefully degrades to CPU inference using GGUF-quantized models (Llama 3 8B Q4_K_M).

🔒 Local LLM Privacy — No External API Calls

For UAE clients with strict data privacy requirements, we deploy Llama 3 8B/70B or Qwen-72B on Azure confidential VMs (DCasv5 or ECasv5 series). These VMs run inside your private VNet, use AMD SEV-SNP encrypted memory, and make no external API calls. Inference latency is comparable to Azure OpenAI for models of equivalent size. No data is ever sent to Microsoft's public OpenAI endpoints.

Production / Enterprise Configuration

The components below apply to the Production (Cloud) and On-Premises tiers. They are not provisioned for the MVP/Pilot, which runs on managed serverless containers.

🕹️ Control Plane — High Availability (3–5 Nodes)

  • etcd cluster with odd-node quorum (3 or 5 nodes) for leader election and state persistence. Automated snapshot backups every 15 minutes to object storage.
  • Cloud: Azure Kubernetes Service (AKS) — Microsoft-managed control plane, automatic upgrades, integrated Azure AD RBAC.
  • On-Premises: Rancher RKE2 or kubeadm with keepalived VIP for API Server HA. OIDC integration with client LDAP / Active Directory.
  • OPA Gatekeeper enforces policy: GPU node scheduling is restricted to authorised AI namespaces; prevents accidental CPU workload placement on GPU nodes.

🎮 GPU Worker Node Pool

  • NVIDIA GPU Operator (single Helm chart) manages the full GPU software stack: driver, CUDA toolkit, container runtime (nvidia-container-toolkit), device plugin, and DCGM exporter — eliminating manual driver lifecycle management.
  • NVIDIA Device Plugin exposes nvidia.com/gpu as a schedulable Kubernetes resource. Resource requests enforced at Pod spec level.
  • Node Taints & Labels: nvidia.com/gpu=present:NoSchedule — only Pods with matching toleration land on GPU nodes. Prevents resource contention from non-AI workloads.
  • MIG (Multi-Instance GPU) Partitioning on A100/H100: up to 7 isolated MIG slices per GPU, each with dedicated compute, memory bandwidth, and L2 cache — enabling multiple inference services to share one physical GPU with performance isolation.
  • Time-Slicing on RTX 5090 / T4 / V100 (Production deployment): multiple Pods share GPU compute in time-multiplexed fashion for workloads that don't require MIG isolation.

💻 CPU Worker Node Pool — Applications & Orchestration

  • API Gateway: Kong / NGINX Ingress with rate limiting, JWT validation, and API versioning. TLS termination at ingress layer.
  • MDM Dashboards & NL Interface: React frontend served via Nginx pods; WebSocket connections for real-time KPI streaming.
  • Workflow Engine: Apache Airflow on Kubernetes (KubernetesExecutor) — DAG-driven ETL, data-quality checks, and cross-system orchestration triggers.
  • Data Integration Services: Kafka consumers, Oracle CDC connectors (Debezium), ERP/CRM REST adapters — all stateless Deployments with HPA.
  • Metadata & Governance: Apache Atlas or DataHub as StatefulSets with ZooKeeper/Kafka backing for lineage, classification, and policy enforcement.

📐 Autoscaling — HPA Based on MDM Query Load & Queue Depth

  • Horizontal Pod Autoscaler (HPA): Scales NLP inference replicas when Kafka consumer lag (queue depth) exceeds threshold, or when P95 inference latency > 900 ms — ensuring SLA adherence during peak query bursts.
  • KEDA (Kubernetes Event-Driven Autoscaling): Zero-to-N scaling for anomaly detection consumers and document extraction workers driven directly by Kafka topic lag. Scale-to-zero at off-peak hours to reduce GPU cost.
  • Vertical Pod Autoscaler (VPA): Right-sizes memory requests for ETL and analytics pods based on observed runtime profiles — prevents OOM kills without over-provisioning.
  • Cluster Autoscaler (Cloud): Provisions / deprovisions Azure GPU VM nodes based on pending Pod resource requests — cost-optimal elastic GPU capacity for bursty AI workloads.

💾 Storage — CSI Drivers

  • Block Storage (PostgreSQL, model checkpoints): Azure Disk CSI (Premium SSD P40) for cloud; Rook-Ceph RBD or NetApp Trident for on-premises. ReadWriteOnce volumes.
  • Object Storage (documents, unstructured data, logs): Azure Blob CSI (NFS mount) for cloud; Ceph RGW or MinIO for on-premises. S3-compatible API. ReadWriteMany volumes.
  • StorageClasses: gpu-fast (NVMe block, RWO) · data-lake (S3-compatible, RWX) · archive (cold blob, RWX).

📊 Monitoring — Prometheus + Grafana + DCGM

  • Prometheus + Grafana: Full-stack metrics (node, pod, container, application). Pre-built MDM dashboards for query latency, data-quality scores, pipeline SLAs, and AI agent throughput.
  • NVIDIA DCGM Exporter: GPU utilisation, memory bandwidth, temperature, NVLink throughput, and MIG slice health — surfaced in dedicated GPU performance dashboards.
  • Loki + Promtail: Centralised log aggregation across all MDM microservices, inference logs, and ETL audit trails.
  • Alertmanager: PagerDuty / Microsoft Teams / email routing for GPU saturation, pipeline failures, and data-quality threshold breaches.

🚀 CI/CD & AI Model Registry — ArgoCD GitOps

  • ArgoCD (GitOps): All Kubernetes manifests and Helm charts versioned in Git. Automated health checks and sync for every MDM service — declarative, auditable deployments.
  • Container Registry: Azure Container Registry (cloud) or Harbor (on-premises) with Trivy vulnerability scanning and Cosign image signing.
  • Model Registry: MLflow tracks all AI agent experiments, parameters, and artefacts. Staged promotion: dev → staging → production via ArgoCD ApplicationSets.
  • NVIDIA Triton Inference Server: Serves NLP, forecasting, and anomaly models via gRPC/REST with A/B model routing for zero-downtime model updates.
  • Automated Retraining: Airflow DAGs trigger weekly GPU retraining jobs when the anomaly agent detects data drift — closed-loop model freshness without manual intervention.
