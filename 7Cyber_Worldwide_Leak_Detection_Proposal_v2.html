The Problem This Module Solves

Clients need a fast, evidence-backed answer to "what of ours is already exposed?", without manually querying six different intelligence tools themselves. The module lets a client submit either a domain or an uploaded file, dispatches that input across six specialised external intelligence services running as five parallel jobs, aggregates, deduplicates and normalises the findings as the services classify them, applies the specified 0–100 weighting, and presents the result as a report-aligned dashboard the client can export. Once enrolled, the scan re-runs automatically every month with a new-vs-resolved delta, and every run is retained with its execution record.

🔁

Monthly Monitoring

An automated scheduler re-runs each enrolled client's scan monthly, computes the delta against the prior run, and raises email alerts for new findings meeting the agreed critical-alert criteria. Every run is retained with its execution log.

Every finding, scan, and log entry is scoped to a tenant record and a tenant data path, with no cross-tenant visibility. Integration configuration is tenant-specific; actual credentials live in Secret Manager.

Open ingestion APIAn HTTPS endpoint accepting syslog-formatted messages (RFC 5424, best-effort RFC 3164) via authenticated POST. Each client's SIEM (or a lightweight forwarder such as rsyslog, Fluent Bit, or NXLog configured to forward over HTTPS) sends its log stream here.

Durable bufferingA Pub/Sub queue between the API and processing, so a burst of messages is absorbed rather than dropped or timed out.
AI-assisted parsingEach message is sent to an LLM with instructions to extract a defined set of structured fields: timestamp, host or device, severity, event category, source/destination IP where present, user where present, and a short action summary. Messages the model cannot parse with reasonable confidence are stored raw and flagged, not dropped.
Structured & raw storageBoth the raw message and the parsed structured record are retained, tenant-scoped, on the same retention terms as the rest of the module (C14).

Admin panel extensionGenerate and revoke each client's ingestion token, and see basic ingestion health: message count, last-received time, and parse success rate.
❌ What This Doesn't Include in v1 (Honest List)
  • No raw syslog over UDP/TCP port 514. Ingestion is HTTPS-only. This keeps the ingestion endpoint serverless and autoscaling, consistent with the rest of the platform, rather than requiring a persistent listener process. Most modern SIEMs and log forwarders support HTTPS forwarding natively; where a client's SIEM only speaks raw syslog, a small forwarder such as rsyslog or Fluent Bit sits in front of it to convert the transport, at the client's side, not ours.
  • No per-vendor custom parsers. One general-purpose AI parsing step handles common formats reasonably well. It is not guaranteed to correctly structure every proprietary or unusual vendor format. Consistently unparseable formats from a specific SIEM are a follow-up tuning item, not a v1 commitment.
  • No correlation with the six intelligence-service findings. SIEM events and scan findings are shown as separate dashboards in v1. Cross-referencing them is a distinct analytical capability, not included here.
  • No alerting rules on ingested events. v1 is dashboards only. "Notify us when X pattern appears in the SIEM feed" is a separate, priceable addition once the alerting rules are defined.
  • No historical backfill. Only messages sent after go-live are ingested. Importing a client's existing log archive is a separate, volume-dependent piece of work.
  • One-way only. The platform receives logs; it does not write anything back to the client's SIEM.

This is the same honesty standard as the rest of this proposal: everything above is well defined and separately priceable once 7Cyber has real usage data to justify it.

📊 Why It's Priced This Way

  • This is real new architecture, priced like one. Unlike the dashboard and admin-panel additions in the previous round, which reused existing patterns, an ingestion pipeline with AI-assisted parsing is a new component end to end: a new public-facing API surface, a new buffering layer, a new data model, and a new integration with an LLM provider. 94 of the 120 hours are development, and most of that is Lead-level architectural work, not routine CRUD.
  • The AI parsing step is the single largest task (22 hours) for a reason. Getting a reliable, bounded JSON schema out of an LLM across varied and messy real-world syslog formats, with sane fallback behaviour when it fails, is the highest-uncertainty part of this add-on and the part most likely to need iteration.
  • We kept the same discipline as the rest of this proposal. No raw UDP/TCP listener, no per-vendor parsers, no correlation engine, no alerting rules, no historical backfill — each would meaningfully add scope, and none were asked for. Adding them later is a scoped, priced change request once 7Cyber knows which of them actually matter in practice.
  • The timeline extension is real, not padded. The Lead Developer is already the critical path at 200 hours over 5 weeks. Adding 64 Lead-level hours to that role, rather than quietly pushing AI-parsing and pipeline architecture onto the Support Developer, is what an honest estimate looks like. It is why this add-on moves delivery from 5 weeks to 6.6.
  • Every rate is the same named Rate Card rate used throughout this proposal. No premium for the AI-integration work. Both developers are priced at Middle Software Engineer, QA at Middle Manual QA Engineer, PM at Project Manager.

Technical Architecture

A React portal with a role-restricted admin panel, a stateless serverless backend, a document store, and an object store for raw payloads.

Layer 1: React Portal (Client-Facing)

  • /scan: landing page with the client's scan history and a new-scan button.
  • /scan/new/domain and /scan/new/file: the two input forms.
  • /report/<scan_id>: the thirteen-panel dashboard, rendered from stored scan data, exportable via the browser's print-to-PDF.
  • Firebase Authentication for login. Firestore security rules enforce tenant isolation on every read, and the client SDK operates strictly under those rules.

Layer 2: Async Processing (Cloud Tasks)

  • Scan submission queues five parallel Cloud Tasks covering the six intelligence services. FOFA and Shodan share one network-exposure task.
  • The aggregation function runs once all tasks report back or time out. It normalises, deduplicates, scores, and persists the results.
  • 30-second per-service timeout, 3 retry attempts through Cloud Tasks' built-in retry policy. An individual service failing does not abort the scan; partial results are valid, and each source run records its own status.
  • If all five tasks fail, the scan is marked failed and the client is notified.
  • Cloud Tasks autoscales, so there is no fixed worker pool to size or manage.

Layer 3: External Intelligence Integration

  • Six adapters, each wrapping its own auth style: API-key header, OAuth2 token, API key plus email parameter, or no auth at all (tgstate).
  • The domain path calls all six services directly. The file path routes Vision-extracted entities through the identical set of calls, sharing one aggregation code path.
  • Each adapter maps the service's response into the shared finding model, preserving the severity and classification the service returns. No classification, correlation, or severity-derivation logic is built on top of it beyond the weighting defined in the specification.
  • Integration configuration is tenant-specific and stored in Firestore as Secret Manager references. Actual credential values live in Secret Manager and are resolved server-side, never exposed to the browser or written to logs.

Layer 4: Data Layer (Firestore + Cloud Storage)

Everything runs inside 7Cyber's own GCP project. Structured scan data and execution metadata live in Firestore; large raw vendor payloads, where retention is required, go to Cloud Storage and are referenced from the corresponding source-run record.

  • Structured scan history and execution records are retained for the operational period agreed with 7Cyber. Raw vendor payload retention is set through a Cloud Storage lifecycle rule and confirmed before go-live (C14).

Layer 5: 7Cyber Admin Panel

  • Role-restricted routes under /admin inside the same React application, visible only to accounts holding the 7Cyber admin role.
  • Client list and users, the client's plan with its allowed monitoring targets and remaining scan quota, monitored domains, and monthly-monitoring on/off per client.
  • Next-scan date per domain, an on-demand re-run guarded against running before the scheduled date or beyond the remaining quota.
  • Scan history per client with score, delta, and status. Any prior run can be reopened. Per-source results show which services returned data and which timed out or errored.
  • Technical log review per run, and JSON export of results and logs as a download or through an authenticated API endpoint. Automatic forwarding to an external SIEM or webhook is not included (see C11).

Layer 6: Dashboard & Export

  • Thirteen panels mapped to 7Cyber's report examples, listed in full in the Scope section.
  • Chart.js for the gauge, bars, donut, and trend line; D3.js for the risk heat map; Leaflet.js for the global exposure map.
  • The client exports the on-screen dashboard to PDF via the browser's native print function. There is no server-side rendering component to build or maintain.

Layer 7: Security, Logging & Retention

  • Firebase Authentication issues the identity; Firestore security rules enforce per-tenant scoping and separate the client-user role from the 7Cyber admin role. Every request is authenticated and scoped to a tenant.
  • External service credentials are held in Secret Manager and resolved server-side. The runtime service account receives only the roles it needs, and secret access is scoped to the specific secrets used.
  • Scan runs, per-source execution results, and admin actions are retained as structured records, giving 7Cyber a retained operational and execution history per client. This is a retained operational history, not a searchable audit-trail system, as stated in the scope exclusions above.
  • We recommend a focused penetration test before go-live, scoped separately from this build. Production go-live follows its successful completion.

Layer 8: SIEM Log Ingestion (New Add-On)

  • A dedicated ingestion endpoint accepts authenticated HTTPS POST requests carrying syslog-formatted messages, one bearer token per tenant, generated and revocable from the admin panel.
  • Accepted messages publish to a Cloud Pub/Sub topic before anything else happens, so a burst of traffic from a client's SIEM is buffered rather than dropped or timed out at the API.
  • A subscriber Cloud Function drains the topic, writes the raw message to Firestore, and calls the LLM API (C16) with a fixed schema to extract structured fields. A message the model cannot parse with reasonable confidence is stored raw and flagged, not discarded.
  • Structured and raw SIEM events are tenant-scoped, on the same Firestore security-rule and retention model as the rest of the platform.
  • Three dashboard panels (Event Volume Over Time, Severity Breakdown, Top Event Sources/Types) render client-side from the structured event store, the same pattern as the other thirteen panels.
