Worldwide Leak Detection Module

A React portal that scans a domain or file across six intelligence sources, presents the results in dashboard views aligned with 7Cyber's report examples, gives 7Cyber's team an admin panel to run the service, and can export scan results to a client's SIEM in four standard formats. Built on the same lightweight serverless backend as the previous version.

Rescoped Around 7Cyber's Feedback

7Cyber reviewed the previous version and raised three points: the platform's dashboards should follow the structure of 7Cyber's existing reports, 7Cyber's own team needs an admin panel, and scan history and technical logs must live in a cloud database with a JSON export path. Findings arrive already classified from the intelligence services; the platform receives, stores, and visualises them. That boundary is what keeps this increase contained. The corrected concept, confirmed by 7Cyber and detailed in a written specification, is scoped, costed, and clearly marked as its own addition in a dedicated section below.

The Problem This Module Solves

Clients need a fast, evidence-backed answer to "what of ours is already exposed?", without manually querying six different intelligence tools themselves. The module lets a client submit either a domain or an uploaded file, dispatches that input across six specialised external intelligence services running as five parallel jobs, aggregates, deduplicates and normalises the findings as the services classify them, applies the specified 0–100 weighting, and presents the result as a report-aligned dashboard the client can export. Once enrolled, the scan re-runs automatically every month with a new-vs-resolved delta, and every run is retained with its execution record.

Client uploads a Vision-supported file (JPEG, PNG, WEBP, PDF, or TIFF). Google Vision API extracts text and entities first; those drive the same six-service lookup.

🔁

Monthly Monitoring

An automated scheduler re-runs each enrolled client's scan monthly, computes the delta against the prior run, and raises email alerts for new findings meeting the agreed critical-alert criteria. Every run is retained with its execution log.

🔒

Tenant-Isolated by Design

Every finding, scan, and log entry is scoped to a tenant record and a tenant data path, with no cross-tenant visibility. Integration configuration is tenant-specific; actual credentials live in Secret Manager.

❌ What This Doesn't Include (Honest List)
  • No BigQuery analytics warehouse. The exposure-over-time trend is computed from retained scan history in Firestore. A warehouse becomes worthwhile only at higher client volumes, and can be added later without changing the data model.
  • No server-rendered, pixel-matched 6-page PDF. The client exports the on-screen dashboard through the browser's print-to-PDF. A custom-rendered branded document remains a priced add-on.
  • No AI-generated CEO Summary. The summary panel uses templated text driven by the risk score and top findings, not an LLM.
  • No Privacy & Regulatory Alignment table and no E7 Recommendation Scorecard. These two panels appear in 7Cyber's report examples but require a rules engine mapping findings to regulatory obligations. We propose scoping them separately once the mapping logic is agreed.
  • No manual analyst review of findings. All findings come from the six automated services.
  • No in-house classification or correlation engine. Severity and classification arrive from the intelligence services. The platform maps, normalises, deduplicates, and applies the weighting already defined in the specification. It does not derive severity from raw data or correlate findings across sources beyond that model. This was confirmed with 7Cyber and is the single largest reason the increase over the previous version is modest.
  • No operations console. No source-health dashboard, alert-rule designer, retry-management screen, or audit-trail search in the admin panel.
  • No subscription purchase or billing flow. Plans and quotas are assigned administratively by 7Cyber. Client-facing plan selection, payment processing, invoicing, renewals, and self-service subscription changes are excluded and quoted separately (see C12).
  • No SIEM-side receiver hosting. The portal delivers to a host:port the client names, or makes a file available; standing up or managing the receiving syslog server or SIEM instance itself is the client's own infrastructure.
  • No simultaneous multi-format SIEM sending. One format is active at a time, per 7Cyber's own specification.

Everything on this list is well defined and separately priceable. Nothing here blocks delivery of the core module as scoped, subject to confirmation of the assumptions listed below.

✚ New — Corrected Following 7Cyber's Confirmed Specification

SIEM Export — Push or Pull, Four Formats

An earlier draft of this proposal described the wrong concept for SIEM integration and has been removed entirely. 7Cyber has since provided and confirmed a detailed specification: after each of the six intelligence services returns results, the portal converts a copy of that data into one of four formats and either pushes it to a client-specified syslog receiver or makes it available for the client to pull through the admin panel. The original raw results always remain stored in the portal's database, unchanged, regardless of whether this feature is enabled.

What follows is the most rational build of 7Cyber's own specification for a first release: all four formats, both delivery methods, built on infrastructure already used elsewhere in this platform rather than new infrastructure introduced just for this feature.

🧠 Two Deliberate Engineering Choices

  • No Redis. 7Cyber's specification allows "Redis/Database" for the retry queue. This platform already uses Cloud Tasks for the six-service dispatch pipeline — the exact same pattern fits Push delivery and its retries. Introducing Redis would mean a new piece of infrastructure to provision, secure, and pay for, for no benefit over what's already running.
  • Module logging reuses the existing audit trail. The platform already has an audit_events collection for admin actions. SIEM module actions (enable/disable, format change, sends, errors) extend that same collection rather than standing up a second logging system.
❌ What This Doesn't Include in v1 (Honest List)
  • No simultaneous multi-format sending. Confirmed as out of scope by 7Cyber directly, not a cut we're proposing.
  • No configurable retry policy beyond what's specified. Three attempts, exponential backoff, is built exactly as specified. A fully configurable retry-policy UI is a separate addition if ever needed.
  • No historical backfill. Push and Pull cover data generated after the feature is switched on. Exporting previously-stored scan history retroactively is a separate, one-time, volume-dependent piece of work.
  • No SIEM-side receiver hosting or validation tooling. The portal delivers to the host:port 7Cyber's client names, or produces a downloadable file. Standing up, configuring, or troubleshooting the client's own receiving SIEM is outside this scope.
  • No config-version rollback. A change-history log is kept for auditing, as specified. Reverting to a prior configuration automatically is not included.

This is the same honesty standard as the rest of this proposal: everything above is well defined and separately priceable once 7Cyber has real usage data to justify it.

📊 Why It's Priced This Way

  • 7Cyber's specification did most of the design work for us, and the price reflects that. Unlike a feature we have to design from a rough request, this one arrived fully specified: exact formats, exact field lists, exact retry counts. That removes most of the architectural uncertainty, which is why this add-on costs less than the earlier, incorrect SIEM concept despite covering more ground (four formats and two delivery modes, not one).
  • The breadth is real, so it's still 128 hours, not 50. Four independent format converters, two delivery mechanisms, an admin UI, a retry queue, and delivery logging is genuinely more surface area than any single item added so far. It is spread across Support-level implementation work (the formatters and admin UI) and Lead-level infrastructure work (the two delivery pipelines), matching each task to the level it actually needs.
  • We reused what already exists instead of adding new infrastructure. Cloud Tasks for the retry queue instead of Redis, the existing audit_events collection instead of a new logging system. That discipline is part of why this add-on has no meaningful new recurring cost.
  • The one-week timeline extension is real, not padded. The Lead's 40 added hours (the two delivery pipelines) push the critical path from exactly 5 weeks to exactly 6. The formatters, admin UI, and logging — 44 hours — went to the Support Developer, since that work is well-specified implementation, not open architectural judgment.
  • Every rate is the same named Rate Card rate used throughout this proposal. No premium for the SIEM work. Both developers at Middle Software Engineer, QA at Middle Manual QA Engineer, PM at Project Manager.

Technical Architecture

A React portal with a role-restricted admin panel, a stateless serverless backend, a document store, and an object store for raw payloads.

Layer 1: React Portal (Client-Facing)

  • /scan: landing page with the client's scan history and a new-scan button.
  • /scan/new/domain and /scan/new/file: the two input forms.
  • /report/<scan_id>: the thirteen-panel dashboard, rendered from stored scan data, exportable via the browser's print-to-PDF.
  • Firebase Authentication for login. Firestore security rules enforce tenant isolation on every read, and the client SDK operates strictly under those rules.

Layer 2: Async Processing (Cloud Tasks)

  • Scan submission queues five parallel Cloud Tasks covering the six intelligence services. FOFA and Shodan share one network-exposure task.
  • The aggregation function runs once all tasks report back or time out. It normalises, deduplicates, scores, and persists the results.
  • 30-second per-service timeout, 3 retry attempts through Cloud Tasks' built-in retry policy. An individual service failing does not abort the scan; partial results are valid, and each source run records its own status.
  • If all five tasks fail, the scan is marked failed and the client is notified.
  • Cloud Tasks autoscales, so there is no fixed worker pool to size or manage.

Layer 3: Risk Scoring

  • The weighted 0–100 model applies point values by finding type, exactly as specified, capped at 100.
  • Six category sub-scores and the overall band (Low/Moderate/Moderate-to-High/High) are computed the same way, deterministically, from the same inputs.

Layer 4: Data Layer (Firestore + Cloud Storage)

Everything runs inside 7Cyber's own GCP project. Structured scan data and execution metadata live in Firestore; large raw vendor payloads, where retention is required, go to Cloud Storage and are referenced from the corresponding source-run record.

  • Chart.js for the gauge, bars, donut, and trend line; D3.js for the risk heat map; Leaflet.js for the global exposure map.
  • The client exports the on-screen dashboard to PDF via the browser's native print function. There is no server-side rendering component to build or maintain.

Layer 7: Security, Logging & Retention

  • Firebase Authentication issues the identity; Firestore security rules enforce per-tenant scoping and separate the client-user role from the 7Cyber admin role. Every request is authenticated and scoped to a tenant.
  • External service credentials are held in Secret Manager and resolved server-side. The runtime service account receives only the roles it needs, and secret access is scoped to the specific secrets used.
  • Scan runs, per-source execution results, and admin actions are retained as structured records, giving 7Cyber a retained operational and execution history per client. This is a retained operational history, not a searchable audit-trail system, as stated in the scope exclusions above.
  • We recommend a focused penetration test before go-live, scoped separately from this build. Production go-live follows its successful completion.

Layer 8: SIEM Export — Push & Pull (New, Corrected)

  • On completion of an intelligence-service run, the aggregation function's stored result is available to the SIEM export module as a read-only source; the original record is never modified.
  • Push: when enabled, a Cloud Task is queued per new result. The task converts the record with the tenant's active IEventFormatter, opens a TCP or UDP connection (TLS for TCP where configured) to the tenant's configured host:port, and sends it. Failure triggers up to 3 retries with exponential backoff; every attempt writes to siem_delivery_log.
  • Pull: an on-demand "Generate" action, or an optional daily Cloud Scheduler trigger, collects the period's results, converts them with the same formatter, and packages them into a ZIP. On-demand downloads serve directly through the authenticated portal; scheduled generation writes to a protected Cloud Storage path with a signed URL valid for 24 hours.
  • All four formatters (Structured JSON, CEF, Syslog RFC 5424, Syslog RFC 3164) share the same IEventFormatter interface, use UTC timestamps throughout, and apply RFC/CEF-correct character escaping.
  • The global toggle is off by default; when off, no conversion or send action runs, regardless of saved configuration.
