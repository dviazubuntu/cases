Section 06
Migration Approach & Tooling
🔧
Confirmation: no licensed third-party migration tool is required or included in this estimate. Migration is implemented entirely through Power Automate flows and, where throughput demands it, an optional Azure Function accelerator. No SharePoint Migration Tool (SPMT) or commercial tooling license is needed — this keeps the fixed price free of third-party licensing exposure.
PriorityToolScope
PrimaryPower Automate flows (SharePoint Move File, Send HTTP Request to Graph API)Full migration orchestration: enumerate source, copy files, re-apply metadata/Term Store, assign Superseded status, log outcomes
Performance accelerator (optional)Azure Function (C# or Node.js, triggered by Power Automate HTTP action)Offload high-volume or CPU-intensive operations (bulk Graph API batching, large folder enumeration, parallel file moves) that would otherwise hit Power Automate action limits or timeouts
ValidationPowerShell (PnP.PowerShell module) — freePre/post reconciliation scripts: document count, metadata, permissions, and Term Store checks
FallbackManual batch move with metadata re-taggingSmall-volume edge cases or items that fail automated processing
Power Automate as primary migration engine
  • Uses the SharePoint connector (Move File action) or Graph API calls (Send HTTP Request) for file relocation
  • Full metadata and Term Store values preserved by explicitly writing column values after move, avoiding known SPMT Term Store drop issues
  • Folder hierarchy recreation handled step-by-step in the flow logic before moving files
  • Superseded status assigned inline as part of the same flow run — no separate post-migration step needed
  • All runs logged to a SharePoint list or Azure Table for reconciliation and audit
Azure Function accelerator (optional — activates only if needed)
  • Triggered by Power Automate via HTTP action; results returned asynchronously
  • Enables parallel batch processing of large libraries that exceed Power Automate's 5,000-action or 30-day run limits
  • Can batch Microsoft Graph API calls (up to 20 per $batch request) for significantly faster enumeration and metadata writes
  • Runs on an Azure consumption plan — no standing infrastructure cost; pay-per-execution only
  • Decision to activate is made during Phase 1 assessment based on document volume and folder depth
💰
If activated, the Azure Function accelerator is an implementation choice within the existing fixed price — it addresses how migration is executed, not a change to what is delivered. It requires an ARO Drilling Azure subscription with at least Contributor access to deploy a Function App; this is confirmed with ARO Drilling in Phase 1 (see Section 13, Dependencies) and consumption-plan Azure hosting costs, if any, are billed directly to ARO Drilling's own Azure subscription, not to SoftTeco's fixed price.
Constraints and implications
  • The Power Automate SharePoint connector is subject to throttling; flows implement pagination, exponential backoff retries, and off-peak scheduling
  • No native rollback: a pre-migration snapshot (file count + metadata export) is taken before each wave; reconciliation scripts detect gaps
  • An Azure Function requires an Azure subscription with at minimum Contributor access to deploy a Function App — confirmed with ARO Drilling in Phase 1
Section 07
Storage Optimization Approach & Platform Limitations
Storage reduction mechanisms (within the fixed price)
  • Migrating inactive/historical content from the active NBP site collection to a dedicated Archive Site logically separates storage buckets
  • Version cleanup automation (Workflow 3) removes versions beyond the latest 3 in K1–K4, directly reducing storage consumed by version history
  • The combination of both is expected to materially reduce active site storage utilization toward the 25 TB threshold identified in the SOW
⚠️
Important platform clarification. SharePoint Online storage is pooled per tenant, not per site. Moving content to the Archive Site reduces active NBP site storage quota usage, but does not by itself reduce total tenant-wide storage consumption, since the Archive Site draws from the same tenant pool. This is a Microsoft 365 platform characteristic, not a limitation of the proposed solution — see the platform limitations table below.
Microsoft 365 / SharePoint architectural limitations
LimitationImpactMitigation

Option: Azure Blob Storage cold tier as long-term archive layer

If SharePoint tenant storage remains under pressure after the Archive Site migration, documents can be offloaded further to Azure Blob Storage (Cool or Archive access tier), which physically removes content from the M365 storage quota.

AspectDetail
Target tierAzure Blob Storage — Cool tier (infrequently accessed, millisecond retrieval) or Archive tier (rarely accessed, rehydration required before download)
Storage costCool: ~$0.015/GB/month; Archive: ~$0.002/GB/month — significantly cheaper than a SharePoint Online storage add-on
Integration approachPower Automate triggers an Azure Function that calls the Azure Blob Storage REST API to upload the file binary and writes blob metadata tags to preserve SharePoint column values
Metadata preservationKey metadata stored as Azure Blob metadata tags and in a SharePoint index list (Document Name, Archive Date, Original Library, Term Store values, superseded status)
RetrievalUsers search the index list in SharePoint; files retrieved on-demand via Azure Function or direct blob URL (with SAS token, scoped access)
