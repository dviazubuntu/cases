From a Broad AI Portal to a Focused First Release — What Changed

The original AI Portal proposal (v4.1, May 2026) scoped eight modules on a different platform architecture, priced at $130K / ~2,600 hrs / 7–8 weeks. Since then, 7Cyber ran its own discovery and sent us a complete technical specification for one module — Worldwide Leak Detection — to be built first, as a native module inside 7Cyber's existing Odoo instance. 🧱 Native Odoo Build

Building as an Odoo module means multi-company (tenant) isolation, RBAC, the portal shell, authentication, and the admin back-office are largely inherited from Odoo itself rather than built from scratch. This meaningfully reduces platform-plumbing effort versus a standalone build.

🎯 Narrower, Sharper Scope

One module instead of eight means a smaller team, a smaller price, and a shorter path to a production-ready release — with the eight modules from the original proposal remaining available as a follow-on roadmap once this module is live and generating value.

💰 Lean, Senior-Led Team

Beyond narrowing the scope, we re-checked the team plan role by role and consolidated overlapping responsibilities onto senior generalists — see Section "Module Effort & Cost" for the leaner, senior-led team and price this produced.

The Problem This Module Solves

Clients need a fast, evidence-backed answer to "what of ours is already exposed?" — without manually querying six different intelligence tools themselves. The Worldwide Leak Detection module lets a client submit either a domain or an uploaded file/photo, dispatches that input across six specialised external intelligence services, run as five parallel integration jobs, aggregates, deduplicates, and normalises the findings, computes a 0–100 digital risk score, and renders the result as an interactive dashboard and a downloadable PDF — matching the visual format of 7Cyber's existing E7 Cybersecurity report. Once a client is enrolled, the scan re-runs automatically every month with delta tracking, so new exposure is caught without the client having to ask again.

🌐

Path A — Domain Input

Client enters a domain (e.g. company.com). The system dispatches direct, parallel requests to all six intelligence services — no pre-processing required.

📎

Path B — File / Photo Upload

Client uploads a Vision-supported file (JPEG, PNG, WEBP, PDF, or TIFF). Google Vision API extracts text (OCR), objects, and logos first; the extracted domains, emails, and company names then drive the same six-service lookup.

📊

One Score, Six Sources

Intelligence X, DarkOWL, FOFA, Shodan, tgstate, and Grayhatwarfare findings are normalised and weighted into a single 0–100 Digital Risk Score with category sub-scores.

📄

Dashboard + PDF, E7 Format

Results render as a live portal dashboard (gauges, heat map, choropleth, donut chart) and a 6-page branded PDF matching the existing E7 Cybersecurity report design.

🔁

Monthly Monitoring

An automated scheduler re-runs each enrolled client's scan on the 1st of every month, computes a delta against the prior scan, and alerts on new Critical findings.

🔒

Tenant-Isolated by Design

Every finding and report is scoped to a company record and a tenant-specific data path — no cross-tenant visibility, consistent with 7Cyber's multi-client model. Integration configuration is tenant-specific, while actual credentials are stored securely in Google Secret Manager.

Technical Architecture

A single Odoo module with an async processing core, two purpose-built Google Cloud data stores, and a reporting layer that renders the same data to a live dashboard and a downloadable PDF.

Layer 1: Odoo Portal Layer

  • /my/leak-scan — client portal landing page: scan history list, new-scan button.
  • /my/leak-scan/new/domain and /my/leak-scan/new/file — the two input forms, the latter triggering the Vision pre-processing step.
  • /my/leak-report/<scan_id> — the live dashboard, rendered from Firestore data.
  • /my/leak-report/<scan_id>/pdf — PDF download endpoint, serving the stored ir.attachment.
  • Standard Odoo authentication and multi-company (tenant) isolation — no separate auth system to build.

Layer 2: Async Processing — queue_job

  • Parent job — scan_dispatch: creates five parallel child jobs covering the six intelligence services (FOFA and Shodan share one network-exposure job) and waits for all to complete.
  • Child jobs (parallel): fetch_intelligence_x, fetch_darkowl, fetch_fofa_shodan, fetch_tgstate, fetch_grayhatwarfare.
  • Aggregation job: runs once all child jobs complete — normalises, deduplicates, scores, persists to Firestore + BigQuery, generates the report.
  • Notification job: emails an alert whenever a scan produces Critical findings.
  • Per-API timeout of 30 seconds, exponential backoff, 3 retry attempts; an individual source failing does not abort the scan — partial results are valid and each finding stores a fetch_status (success / timeout / error). If all integration jobs fail, the scan is marked failed and the client is notified. All jobs log to queue.job for admin visibility.

Layer 3: External Intelligence Integration

  • Six independent adapters (one per source), each wrapping its own auth style — API-key header, OAuth2 token, API key + email param, or no auth at all (tgstate).
  • Domain path calls all six services directly; file path routes Vision-extracted entities (domains, emails, phone numbers, company names) through the identical set of service calls — one shared aggregation code path for both input types.
  • Per-tenant integration config (leak.api.config) holds Secret Manager references, not raw credentials; actual secret values live in Google Secret Manager and are resolved server-side. Credentials never appear in frontend responses or logs; failed auth triggers an admin notification.

Layer 4: Data Layer — Two-Tier Google Cloud Strategy

  • Firestore — real-time scan data. Individual findings, scan status, and the dashboard JSON payload, queried server-side by the authenticated Odoo controller for live rendering (the browser never accesses Firestore directly). Path: /tenants/{company_id}/scans/{scan_id}/findings/.
  • BigQuery — analytics and history. Aggregated monthly results, country-level exposure (global map), time-series exposure trends (6-month chart), and finding-type distributions. Partitioned by scan_date for efficient deletion and cost management.
  • Monthly delta object tracked per monitor: new_findings, resolved_findings, persisted_findings, and risk_score_delta (positive = worsened, negative = improved).

Layer 5: Reporting & Visualisation

  • Dashboard panels: Digital Risk Score gauge, Risk Scores by Category bar chart, Risk heat map (likelihood × impact), Findings by Source donut, Exposed Data Types list, Data Privacy Impact gauge, Executive Exposure Status table, Infostealer Intelligence panel, Top Risk Cases cards, Privacy & Regulatory Alignment table, E7 Recommendation Scorecard, Global Exposure Map, 6-month Exposure trend line, CEO Summary panel.
  • 6-page PDF: Cover → Executive Summary → Data Exposure Summary → Global Exposure Map → Risk Analysis → Intelligence Details. Auto-generated on scan completion; clients can re-download any historical PDF from the scan-history list.

Layer 6: Security, Compliance & Retention

  • Firestore is accessed exclusively by the authenticated Odoo backend. Tenant isolation is enforced in the Odoo controller and model access layer before any Firestore operation; the Firestore service account is restricted through IAM on a least-privilege basis, and the browser never accesses Firestore directly. Every Odoo model carries company_id and is filtered by record rules; PDF attachments use non-guessable access tokens.
  • Scan findings retained indefinitely as history, per the spec, with a delete_scan_history action available on request; BigQuery partitioning supports efficient, targeted deletion.
  • All external service credentials held in Google Secret Manager, encrypted at rest, and never exposed in frontend responses or logs.
  • Given the sensitivity of aggregated breach/credential data, we recommend a focused penetration test of the portal routes before go-live — scoped separately from this build.
