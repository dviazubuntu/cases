Section 01
Executive Summary

ConstructIQ 2.0 is an AI-native construction program management platform designed to take the RAK government construction portfolio from single-project tracking to predictive, portfolio-wide program intelligence. It supersedes the validated interactive POC by grouping related projects under Programs, extending the risk/dependency register into a full RAID log with Actions, broadening resource planning to any non-human resource (equipment, material, plant), integrating natively with Oracle Fusion Cloud ERP for financial and workforce truth, and layering a predictive-then-prescriptive AI analytics engine on top of real tracking data — all while keeping data resident within the UAE.

Program-Native
Every project sits inside a Program from day one. Department Heads and the Director General see a single rolled-up view — with a drill-down into any project's schedule, cost, resource and risk detail — instead of a wall of disconnected project cards.
Predictive, then Prescriptive
Release 1 adds AI-driven schedule and cost forecasting on top of real tracking data. Release 2 adds AI-generated corrective/preventive action recommendations and what-if simulation — moving beyond "a tool that could be done in Excel."
Enterprise-Ready
Custom persona builder, per-project access scoping, Oracle Fusion ERP sync (costing, GL, procurement, AP, HCM), and a reused, already-delivered document control system — all deployed on UAE sovereign cloud infrastructure.

Structured project, schedule, resource, RAID and budget data flows upward through the stack — enriched at each layer — while financial and workforce truth is continuously reconciled against Oracle Fusion Cloud ERP.

Design rationale: Program as a first-class entity
The original POC modelled a flat Workspace → Project hierarchy. Client feedback made clear that a Department Head or the Director General reasons in terms of programs — related projects sharing a client mandate or funding envelope — not a scrolling list of individual cards. ConstructIQ 2.0 therefore inserts Program as a first-class entity between Workspace and Project: Workspace 1—N Program 1—N Project. Every rollup dashboard, risk count, and 4-dimension health score aggregates first at the Program level, then drills down into its constituent Projects.
Data flow
1
A PM or field user creates/updates a project, task, RAID entry, resource booking, or budget line via the web portal or mobile app.
2
The Application & Workflow Layer validates the change against the persona's access scope and persists a structured record, then publishes an event:
{ entity_type, entity_id, project_id, program_id,
  change_type, actor_id, timestamp, payload }
3
The Integration Layer reconciles relevant financial, procurement, vendor, and HR master data with Oracle Fusion Cloud ERP via a secured API gateway (see Section 06).
4
The Analytics Engine recomputes rolled-up KPIs — 4-dimension health, risk counts, budget variance — at both Project and Program level, near real-time.
5
From Release 1, the Predictive layer scores schedule-delay and cost-overrun probability on accumulated historical data (see Section 05).
6
The Dashboard API pushes updated KPI tiles to Program/DG/PM dashboards over WebSocket; the Notification Service fires alerts on threshold breach.
🔒
Data residency: All application data, documents, and AI model artefacts are stored and processed within the UAE, on sovereign cloud infrastructure (Section 10). Oracle Fusion Cloud ERP integration is scoped to the UAE region tenancy; no operational project data leaves UAE borders.

Section 04
AI & Predictive Analytics Pipeline

Per client direction, AI capability is intentionally staged by analytical maturity. Release 1 delivers the data pipeline and predictive forecasting ("what is likely to happen") once real tracking data exists. Release 2 delivers prescriptive recommendations and what-if simulation ("what should we do about it, and what happens if we change X") — explicitly deferred by mutual agreement in the feedback session.

Stage 1 — Release 1
Predictive Analytics Foundation
Time-series & regression models
Forecast outputs:
  • Schedule delay probability
  • Schedule slippage magnitude
  • Cost overrun probability
  • Estimate-at-completion (EAC)
  • Program-level risk trend
Trained on accumulated schedule, budget and RAID data from the platform itself plus reconciled Oracle Fusion actuals — no separate data collection required.
Stage 2 — Release 2
Prescriptive Recommendations & What-If
LLM reasoning · Scenario engine
Corrective action suggestions — proposed responses to a detected risk, cost or schedule signal
Preventive action suggestions — early recommendations before a threshold is breached
What-if scenario simulation — projected cost/schedule impact of a hypothetical change before it is committed
Confidence / explainability indicator — surfaces the contributing factors behind each recommendation
Model foundation & governance
Data requirements for reliable predictions ▼

Predictive accuracy depends on a critical mass of real, tracked project history. The Stage 1 pipeline is deliberately sequenced after the POC and the core Planning & Schedule Control capability (Epic E2) are live and producing genuine schedule/budget/RAID data — not on day one, and not on synthetic data. Historical actuals reconciled from Oracle Fusion (Section 06) supplement the platform's own tracked history for cost-forecast training.

Model lifecycle management ▼

Model registry and experiment tracking (MLflow), scheduled retraining pipelines (Apache Airflow), and staged rollout — new model versions are evaluated against the current baseline before replacing it in production. Accuracy targets and their monitoring cadence are defined in Section 13.

Explainability & adoption trust ▼

Every prescriptive recommendation (Release 2) is accompanied by a plain-language rationale and the underlying metric trend that triggered it, so a PM or Director General can validate — not just trust blindly — each suggestion. This directly addresses the client's stated adoption bar: the platform must visibly do more than "tick mark this, update this."

Section 05
Oracle Fusion ERP Integration

Oracle Fusion Cloud is the system of record for financial and workforce truth. ConstructIQ 2.0 does not duplicate ERP data entry — it reconciles against Oracle Fusion so that budget, procurement and workforce figures shown on any dashboard are trustworthy, not manually re-keyed.

Integration TouchpointScope & Direction
Project Costing syncTwo-way sync of project budgets/actuals with Oracle Fusion Project Management/Costing. Core integration underpinning all budget features.
General Ledger / Chart of AccountsMaps internal cost codes / WBS to Oracle Fusion GL accounts for reconciled budget reporting.
Procurement & Purchase OrdersPulls PO status and commitments from Oracle Fusion Procurement into equipment/material bookings.
Vendor / Supplier MasterKeeps subcontractor and supplier records aligned with Oracle Fusion supplier master data.
Accounts Payable / InvoicesLinks contractual payment milestones to Oracle Fusion AP invoice and payment status.
HR / Workforce Master (HCM)Sources employee, role and availability master data from Oracle Fusion HCM.
Customer / Client MasterAligns client and department records with Oracle Fusion Customer master data.
Identity / SSOFederated authentication via Oracle Identity Cloud, so the platform trusts Oracle's identity provider rather than maintaining a parallel credential store.
Sync monitoring & error logDedicated dashboard for integration health, sync failures and reconciliation exceptions — ships together with the integration itself, not as an afterthought.

Section 07
Security Architecture

Defence-in-depth across all four architecture layers. Persona-based RBAC is a first-class feature, not an afterthought — matching the client's explicit request for self-service persona and project-level access management.

🔐
Data in transit
  • TLS 1.3 for all API traffic
  • WireGuard VPN for gateway-to-Oracle Fusion connectivity
  • Certificate pinning in the mobile app
  • Local LLM fallback mode: zero external API calls
💾
Data at rest
  • PostgreSQL AES-256 encryption at rest
  • Object storage SSE-S3 equivalent for documents
  • HashiCorp Vault for secrets — no credentials in code
  • Encrypted backups with defined retention
🛡️
Access control
  • RBAC enforced at the API gateway layer
  • Custom persona builder + per-project access scoping
  • MFA mandatory for Admin and Director General roles
  • Immutable audit log with defined retention
🇦🇪
UAE data residency
  • All data processed and stored within UAE borders
  • Oracle Fusion Cloud UAE region tenancy only
  • UAE sovereign cloud deployment (Section 10)
  • Client retains full data ownership contractually
Section 08
KPI & Metrics Framework

ConstructIQ 2.0 computes three tiers of metrics, each calibrated to a different stakeholder's decision cycle — directly reflecting the client's own framing of "current state across schedule, cost, resources and partners" plus predictive analytics. All metrics derive from the same underlying tracked data (Sections 04–06) and aggregate upward — no separate data collection is required per tier.

👷
1. Operational Metrics — Project Manager / Field Level
"What is happening on my project(s) right now, and what do I need to act on today?"
Task Status & Workload Drill-down
Definition
Per-person, per-task status: on track, delayed, completed, or overloaded — drilled down from the team FTE allocation view.
Value
Directly answers the client's request to move beyond "just information" on workload into actionable task-level detail.
Equipment & Material Availability
Definition
Booking status and capacity of non-human resources — from handheld tools to cranes, forklifts and power stations.
Value
Prevents double-booking and surfaces upcoming capacity shortfalls before they block a task.
Open RAID Items Assigned to Me
Definition
Risks, Issues, Dependencies and Actions where the current user is the named owner.
Value
A single actionable worklist instead of hunting across separate registers.
Project Budget Burn vs Plan
Definition
Spend-to-date versus planned budget curve at the individual project level, reconciled against Oracle Fusion actuals.
Value
Early warning before a variance becomes a change-order conversation.
📊
2. Program Metrics — Department Head / Program Manager
"Which projects in my program need attention, and how are resources allocated across it?"
4-Dimension Program Health Rollup
Definition
Aggregated Schedule, Cost, Resources and Partners health across every project in the program.
Value
The exact four dimensions the client named as the basis of current-state analytics.
Risk Count Rollup with Drill-down
Definition
The number — not colour — of open risks per project, rolled up at program/department level, drillable into the contributing project(s).
Value
Replaces the single RAG dot the client explicitly said "doesn't really help."
Cross-Project Resource Utilisation
Definition
FTE and equipment allocation viewed across all projects in the program, flagging over-allocation or idle capacity.
Value
Supports staffing and equipment-sharing decisions across the program, not just within one project.
Program Budget Variance
Definition
Combined spend-to-date versus plan across all program projects, with change-order impact called out separately.
Value
Gives the Department Head one number to defend at a program budget review.
🇦🇪
3. Strategic Metrics — Director General
"Is the national construction portfolio on track, and where are the predictive red flags?"
Portfolio-Wide Executive Rollup
Definition
The 4-dimension health index and risk count aggregated across every program in the portfolio, not just a single project.
Value
A true executive view — the client's original request was for a program/portfolio lens, not a single-project dashboard framed as strategic.
Predictive Schedule & Cost Forecast
Definition
AI-forecast probability and magnitude of schedule slippage and cost overrun per program (Release 1).
Value
The predictive lens the client named as his top analytics priority, alongside current state.
Strategic KPI Alignment (RAK Vision 2030)
Definition
Configurable sector KPIs — e.g. delivered capacity, population served, PPP partners active, sustainability rating.
Value
Ties day-to-day project delivery directly to the programme's national strategic mandate.
AI-Recommended Actions Summary
Definition
Portfolio-level digest of open AI-generated corrective/preventive recommendations and their status (Release 2).
Value
The prescriptive layer that answers "does it suggest some corrective or preventive actions?"

Section 10
Implementation Roadmap

Five phases from discovery through continuous optimisation. Effort/complexity is the primary sequencing lens: foundational, low-complexity capabilities ship first; AI-driven predictive analytics is deliberately staged for Release 1, once real tracking data exists; prescriptive AI and what-if simulation are scoped to Release 2, whose detailed estimate follows the Release 1 retrospective.

1
Phase 1 — POC
≤ 1.5 months
73 catalogued POC-tagged features: Program/Project directory and setup, core Gantt scheduling, Team & Equipment registers, base RAID log (Risks, Dependencies, Issues, Assumptions), Project/Executive dashboards, basic access levels, and the reused document management system.
Gate: Client validates POC against the agreed feature list. Sign-off to proceed into Release 1 scoping.
2
Phase 2 — Release 1
≤ 3–4 months
56 catalogued features: task-level scheduling & dependencies, WBS/cost-code budgeting and change orders, full RBAC with custom persona builder, Program-level rollups, RAID governance (Actions, ownership, risk drill-down), and the AI-driven predictive analytics foundation (data pipeline + schedule/cost forecasting). Oracle Fusion ERP integration delivered end-to-end.
Gate: Oracle Fusion UAT passed. Predictive model accuracy baseline established (Section 13). Director-General portfolio dashboard live.
3
Phase 3 — Release 2
Scope TBD — estimated after Release 1
24 catalogued features: AI-generated prescriptive corrective/preventive action recommendations, what-if scenario simulation, custom report builder, portfolio analytics/benchmarking, native mobile app with offline capture, and additional dashboard drill-down filters.
Gate: Release 1 retrospective and predictive-model maturity assessment jointly determine the final Release 2 scope and delivery estimate.
4
Phase 4 — Continuous Optimisation
Ongoing
Monthly AI model retraining cycles with tracked accuracy improvement. Expanding predictive coverage. Recurring roadmap reviews with the client so upcoming features are always visible ahead of time.
Gate: Continuous KPI improvement versus the Release 1/2 baseline, tracked in monthly reviews.
Section 11
Risks & Mitigations

Every major risk identified for this programme has a concrete mitigation built into the phasing and architecture — not a workaround bolted on afterward.

RiskSeverityMitigation
Predictive AI reliability depends on data maturityHighPredictive forecasting is deliberately scoped to Release 1, after POC and core tracking data (Epic E2) is live and generating real history. Accuracy targets and monitoring are defined in Section 13; rule-based alerts remain the fallback until models qualify.
Oracle Fusion integration complexity/timelineHighEarly Fusion API and Chart-of-Accounts discovery in Phase 0–1. Dedicated integration workstream in Release 1. Sandbox validation before production cutover.
Adoption risk — perceived as "just a tracking tool"MediumPredictive and prescriptive AI capability is explicitly on the roadmap and communicated up front (Section 11). POC is scoped to demonstrate task-level drill-down and risk-count rollup value, not only CRUD screens.
Persona/RBAC complexity for a multi-stakeholder government clientMediumCustom persona builder and per-project access scoping ship in Release 1. POC validates a simplified access model with the client early, before the full builder is built.
Data residency / hosting requirementsMediumUAE sovereign cloud (ADG Cloud / G42 Core42) confirmed as the deployment target. Oracle Fusion integration scoped to the UAE region tenancy only.
Scope creep beyond agreed POCLowRelease boundaries are explicitly documented and agreed with the client in this specification. What-if / prescriptive AI is explicitly excluded from POC and Release 1 scope.
Task-level dependency & critical-path accuracyMediumPOC ships the critical-path toggle on existing data; the full dependency engine (finish-to-start and related link types) is built in Release 1 before critical-path output is relied on operationally.
