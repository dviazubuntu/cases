Data Flow & Security

  1. Data Ingestion: Patient enters data via web/mobile/voice → TLS 1.3 encrypted transmission → Validated at API gateway → Temporary storage in Redis (encrypted)
  2. AI Processing: Audio/text sent to GPU cluster within same VPC → STT/NLP/Translation processing → Results returned to application layer → NO external API calls
  3. Data Persistence: Validated data written to PostgreSQL → Encrypted at rest (AES-256) → Documents/images to object storage → Audit log to MongoDB
  4. Backup: Hourly incremental backups to Riyadh → Daily full backups to Jeddah DR → Weekly encrypted tape backup to Dammam vault
Important: Unlike many SaaS providers who route data through US/EU for AI processing, our architecture ensures AI models run locally on KSA infrastructure. This eliminates cross-border data transfers entirely.

5. Kubernetes GPU Management Architecture

Our platform leverages Kubernetes for orchestration, enabling dynamic scaling, fault tolerance, and efficient GPU resource allocation across AI workloads.

Cluster Architecture

🎛️ Control Plane (Dedicated Servers)

Components: kube-apiserver, etcd, kube-scheduler, kube-controller-manager, cloud-controller-manager

Configuration: 3-5 node HA cluster (odd number for quorum), no workloads scheduled on control plane

Purpose: Cluster management, API gateway, state persistence, scheduling decisions

  • etcd: Distributed key-value store for cluster state (encrypted at rest)
  • kube-scheduler: GPU-aware scheduling with custom policies (bin-packing, spread, affinity)
  • Backup: etcd snapshots every 30 minutes to object storage
🖥️ GPU Worker Nodes

Node Pool 1 - AI Inference: Dedicated GPU servers (RTX 5090 or RTX 6000 Ada Pro)

  • NVIDIA Device Plugin: Exposes GPUs as schedulable resources to Kubernetes
  • GPU Operator: Manages NVIDIA drivers, CUDA toolkit, container runtime (nvidia-docker)
  • Node Labels: gpu.nvidia.com/class=RTX5090, workload=ai-inference
  • Taints: gpu=true:NoSchedule (only GPU pods can schedule here)
  • Resource Limits: GPU memory and compute quotas per namespace

Deployed Services:

  • Whisper STT pods (1 GPU per pod, horizontal autoscaling based on queue depth)
  • LLM inference pods (Qwen Medical, 1-2 GPUs per pod depending on model size)
  • TTS pods (Kani Arabic, 0.5 GPU per pod via MPS - Multi-Process Service)
  • NLP pipeline (entity extraction, sentiment, translation)
⚙️ Application Worker Nodes (CPU-only)

Node Pool 2 - General Compute: Standard servers without GPUs

  • Workloads: Web API (FastAPI/Node.js), workflow engine, notification service, caching layer
  • Autoscaling: Cluster Autoscaler + Horizontal Pod Autoscaler (CPU/memory-based)
  • Resource Requests: 2-4 CPU cores, 4-8GB RAM per pod
🔄 GPU Workload Orchestration

Key Technologies:

  • NVIDIA GPU Operator: Automated deployment of NVIDIA software components (drivers, container runtime, device plugin, DCGM for monitoring)
  • Time-Slicing / MPS: Multiple pods share a single GPU when full GPU power not needed (e.g., TTS workloads)
  • Multi-Instance GPU (MIG): For A100/H100 (not RTX), partition single GPU into isolated instances
  • GPU Scheduling Strategy:
    • Bin-packing: Fill nodes before spreading (maximize GPU utilization)
    • Topology-aware: Co-locate NLP pipeline stages on same node to reduce inter-GPU transfers
    • Priority Classes: Voice sessions get higher priority than batch analytics jobs
📊 Monitoring & Autoscaling

Metrics Collection:

  • DCGM Exporter: GPU metrics (utilization, memory, temperature, power) to Prometheus
  • Prometheus + Grafana: Real-time dashboards for GPU health, inference latency, queue depth
  • Custom Metrics: STT requests/sec, LLM tokens/sec, concurrent voice sessions

Autoscaling Logic:

  • HPA (Horizontal Pod Autoscaler): Scale STT pods from 2 to 10 based on request queue (target: <5 second queue time)
  • VPA (Vertical Pod Autoscaler): Adjust memory limits for LLM pods based on actual usage
  • Cluster Autoscaler: Add/remove worker nodes when resource requests exceed/fall below thresholds

Example Scaling Rule: If average GPU utilization >75% for 5 minutes, scale out STT pods by 50%

🔐 Security & Isolation
  • Namespace Isolation: Separate namespaces for ai-inference, app-backend, database, monitoring
  • Network Policies: Restrict inter-pod communication (e.g., LLM pods can't talk to database directly)
  • Pod Security Standards: Enforced baseline/restricted policies (no privileged containers except GPU operator)
  • Secrets Management: External Secrets Operator with Hashicorp Vault for API keys, DB credentials
  • RBAC: Role-based access control for kubectl access, service accounts with minimal permissions
💾 Persistent Storage
  • CSI Drivers: NetApp Trident or Longhorn for dynamic PV provisioning
  • Storage Classes:
    • fast-ssd: NVMe for model weights cache (low latency)
    • standard: SSD for databases
    • archive: Object storage for document/image retention
  • Volume Snapshots: Daily snapshots of all PVCs, retained 30 days
🚀 CI/CD & Model Deployment
  • GitOps: ArgoCD for declarative deployment, Git as single source of truth
  • Container Registry: Harbor (on-premises) for storing Docker images and Helm charts
  • Model Versioning: MLflow or DVC for tracking model versions, A/B testing
  • Canary Deployments: Istio service mesh for gradual rollout (e.g., 10% traffic to new Whisper model)
  • Rollback: Automated rollback if error rate >2% or latency >500ms

Example: Voice Intake Session Flow

  1. Patient initiates voice intake via mobile app → WebSocket connection to API gateway (Nginx Ingress)
  2. API gateway routes to backend pod in app-backend namespace → Creates session ID, stores in Redis
  3. Audio chunks streamed (1-second intervals) → Backend forwards to STT pod in ai-inference namespace
  4. STT pod (GPU-accelerated) transcribes in real-time → Returns text to backend → GPUs auto-scale based on queue
  5. Backend sends text to LLM pod (same namespace) → Generates conversational response + extracts entities
  6. LLM response sent to TTS pod (time-sliced GPU) → Audio generated and streamed back to patient
  7. Session data persisted to PostgreSQL (running on DB worker nodes) → Audit log to MongoDB
  8. Metrics collected: GPU util logged to Prometheus, latency tracked for SLA monitoring

Total Latency Budget: <500ms end-to-end (STT: 200ms, LLM: 150ms, TTS: 100ms, network overhead: 50ms)

Disaster Recovery & HA

  • Multi-Zone Deployment: Control plane and worker nodes spread across 3 availability zones within STC Riyadh DC
  • Pod Disruption Budgets: Ensure minimum 2 replicas of critical services always running during node maintenance
  • Backup Strategy:
    • etcd snapshots every 30 minutes to S3-compatible storage
    • Persistent Volume snapshots daily
    • Full cluster state backup weekly (Velero)
  • DR Site: Standby cluster in Mobily Jeddah DC with daily data replication, can be activated in <4 hours
