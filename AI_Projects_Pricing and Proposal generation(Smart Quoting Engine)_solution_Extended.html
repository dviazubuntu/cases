Scope of the AI-Assisted Proposal Creation Solution

The MVP is deliberately scoped to the proposal creation workflow only — from RFQ intake to a signed-ready quotation in Client's existing format (e.g. Q.4154/WR/RAR/2025/Rev-05 — KAPSARC School of Public Policy). Items outside this perimeter are explicitly deferred to later phases.

✅ In Scope (MVP)

1. RFQ Intake & BOM Extraction

  • Upload of PDF drawings, scanned spec sheets, IFC drawing packages, hardware schedules
  • OCR + LLM-based extraction of item descriptions, dimensions (depth × width × height in mm), materials, finishes, drawing references
  • Manual / semi-automated BOM editor with low-confidence flagging

2. Item Categorisation

  • Classifies BOM lines into product categories: Planter Box, Hospitality Counter, Storage Cabinet, Wall Cladding & Ceiling, Wooden Doors (NFRD), Vanity Counter, Fire House Cabinet, etc.
  • Maps to internal material codes (IWD-01 European Oak, ISSM-02 Acrylic, MR MDF 18mm, etc.)

3. Historical Reference Engine

  • Similarity search across the 70% of past projects with usable records
  • Surfaces past unit prices, suppliers, qty/units, realised margin — for context, not auto-pricing

4. Live Supplier Pricing Capture

  • Structured form for entering today's supplier quotes per BOM line
  • Supplier catalogue with contact, lead-time, and historical quote log
  • Currency = SAR; VAT auto-applied (15%)

5. Single-Item Cost Rollup Engine

  • Composes a complete all-in price per line from five stacked layers: (L1) door / item manufacture, (L2) third-party hardware kit, (L3) logistics, (L4) on-site fixing works, (L5) indirect / VAT / contingency
  • Reproduces the factory cost-sheet logic parametrically (CBM × wood rate, blockboard m² rate, finish, labour, overhead 50%, margin 33%) — same numbers your team already uses in BCC Sedra 5 / Saudi Emar style estimates
  • Resolves the correct hardware set (#1..#8) from door location + type + finish family, pulls live unit prices from the latest supplier quote, and substitutes bronze ⇄ nickel variants automatically
  • Adds logistics and installation labour that the door-factory and hardware-supplier quotes explicitly exclude — so the line price you see is delivered and installed, not ex-factory
  • Every multiplier is auditable; nothing is a black box

6. AI Alternatives Module

  • LLM-driven material / construction substitutions with quantified savings
  • Each suggestion explicitly Approved or Rejected by the estimator

7. Revision Management

  • Version control on every quotation (Rev-01 → Rev-05 lineage)
  • Side-by-side diff between revisions: who changed what, why, and the price impact

8. Document Generation

  • Internal Excel calculation file (full BOM, suppliers, factors)
  • Branded client PDF in Client's existing letterhead — Reference No., date, client, project, location, attention, terms & conditions, notes, signatures (Projects Manager / General Manager)
  • Subtotal, VAT (15%), Total Cost in SAR — matching the current quotation format exactly

9. Approval Workflow

  • Estimator → Projects Manager → General Manager sign-off chain
  • Audit log of every change, approval, and export

Single-Item Cost Rollup — How the AI Builds the All-In Door Price

📋 Committee Evaluation Note

"Kindly review this quotation and evaluate the project from a practical costing perspective, particularly focusing on how an AI-based system can intelligently manage and generate a complete single-item cost structure that incorporates all related components, including door price, hardware, logistics and associated fixing works."

This section answers that question directly, using real numbers from two recent Al-Rashed engagements: BCC Sedra 5 — Villas D2 Ayala & I2 Ayala Hills (34 wooden doors, ES-05 schedule R04 dated 02-04-2026) and Saudi Emar — Kings Roshn Theatre & Joinery (KI26-E898-Q-02-00).

What a "Single Door Line" Really Contains

In your current process, one line in the wooden-doors schedule (e.g. D2-D02 GF Family Toilet — 900 × 2800 — Hardware Set #2 — OAK#50Q veneer) is in fact a stack of five independent cost layers, each owned by a different supplier or team, none of which deliver a single all-in number:

L1 · Door Leaf Manufacture (your factory)

Lumber CBM × wood rate (Hardwood Beech 3,850 / Swedish 1,350 SAR/CBM), blockboard & veneer sheets, factory hardware consumables, varnish/HPL finish, direct labour, 50% overhead, 33% margin. Output: SAR / SQM × door area ex-factory.

L2 · Hardware Kit (3rd-party — e.g. Doors Technology Trading)

5 × concealed hinges (Simonswerk TE 340 3D), lockset (sash / bathroom / hook), cylinder (key or thumbturn), lever / pull handle, optional door closer, escutcheon, stopper. Finish family: nickel mat vs bronze (~30–40% premium). Sold per "set" (Set #1..#8).

L3 · Logistics

Ex-factory delivery to site, lifting, oversize packaging for >2.8 m or sliding tracks, batch consolidation. Explicitly excluded from door cost sheet ("Delivery Ex-Factory only") and from hardware quote.

L4 · Fixing / Installation

Door hanging, frame plumbing & sealing, 5-hinge mounting, lockset & closer fitting, sliding-track alignment, silicone, snagging & touch-up. Explicitly excluded from both supplier quotes ("Installation is not included" / "Installation, fitting, or any associated labor costs are excluded").

L5 · Indirect, VAT, Contingency

Project-level overhead, contingency band, VAT 15% applied per ZATCA. Currently re-keyed manually each revision.

Why this is hard manually today. A single 34-door villa pair (D2 + I2) has 8 hardware sets × 2 finish families × 17 unique size/location combinations. Cost sheets live in one Excel file ("SD NFR HOLLOW VENEER & HPL.xls"), hardware in a supplier PDF (DHQ GY26-0010), schedule in another (ES-05 R04), and fixing/logistics in nobody's quote. When the ID team revises a door size or finish, all five layers must be re-stitched by hand — which is exactly the mechanism that produced Rev-04 then Rev-05 on KAPSARC and forces 1-month estimating cycles.

The Five-Layer Cost Engine — How the AI Composes It

Layer Inputs the AI reads How the AI computes it Output per door line
L1 — Manufacture Size (T × W × L), door type (solid-core / hollow-core / sliding), veneer ref (OAK#50Q, OAK#864Q), wood source, finish (HPL / varnish) Parametric model of your existing cost sheet: CBM rates by component (Lipping / Header / Jamb / Architraves / Rails) × wood-source price; blockboard & veneer sheets × m² rate; factory hardware-materials lookup; 50% overhead on labour; 33% margin. Veneer & HPL costs trigger automatically from the finish field. SAR / SQM × area = ex-factory door selling price
L2 — Hardware Location class (toilet / bedroom / sliding / dirty-kitchen / dining), door type, finish family (nickel mat / bronze) Pattern-matches to a Hardware Set template (#1..#8) using the same rule your team applies (location × door type). Pulls live unit prices from the most recent supplier quotation (e.g. DHQ GY26-0010). Swaps to bronze variants automatically when the door is on the bronze schedule. Set price × set count
L3 — Logistics Project site, batch size, oversize flag (height > 2.8 m, sliding track length) Rule-based estimator — base trucking + per-extra-LM + lift surcharge for oversize. No "AI magic" needed; the value is doing it consistently across every line item every time. Logistics SAR
L4 — Fixing Door type (swing vs sliding vs concealed track), height, hardware complexity (closer? sliding system? fixed panel?), site access Hours × crew rate per door class, with multipliers for sliding tracks, concealed frames, door closers, and 5-hinge hanging. Calibrated against actuals from the past 30 projects where records exist. Fixing labour + on-site materials (silicone, fixings, sealants)
L5 — Indirect / VAT Contract terms, contingency band, current VAT rate Applied as a transparent percentage stack to L1–L4. VAT shown separately on the output document. Final delivered + installed SAR per line

💡 The key claim, in plain language

The AI does not "guess" the all-in door price. It treats the door line as a recipe of five layers — manufacture, hardware, logistics, fixing, indirect — each backed by a transparent calculation that already exists somewhere in Client's process. The contribution of the AI is to wire those layers together, keep them in sync when any input changes, surface alternatives that respect the door schedule, and produce a single delivered & installed SAR figure per line — instead of three Excel files, two supplier PDFs, and a fixing estimate that lives in someone's head.

This is exactly the gap the BCC Sedra 5 estimate and the KAPSARC Rev-05 lineage expose. Closing it is the MVP.

Feasibility — Honest Assessment

Yes, this can be implemented — but with clear-eyed caveats about Client Industrial's process maturity. Pretending the data is cleaner than it is would be the surest way to fail.

Risks to Manage Up-Front

1. No single source of pricing data. Prices come from the market each time. Mitigation: design for fresh supplier input as the primary path; treat history as an assistant, not an oracle.
2. No structured BOM. 3ds Max and AutoCAD don't export specifications cleanly. Mitigation: rely on PDF parsing + a structured web form, not CAD integration.
3. Fully manual culture. Estimators aren't used to working in systems. Mitigation: keep the UX close to the Excel they already know; pilot on a single product type first.
4. Every project is unique. The model can't rely on history alone. Mitigation: hybrid architecture — humans control the inputs that drive the price.
5. Training data is "good enough", not perfect. 70% of projects have usable records; the client accepts manual cleanup. Mitigation: budget BA time for a one-off data cleansing pass before go-live.

The bottom line

Implementation works as a hybrid — AI for parsing, recommendations, and document generation; humans for supplier prices and approvals; and a structured form instead of free-form Excel. Without at least a simple structured input layer, no AI project will succeed here. The recommended approach is to pilot on one product line (e.g. interior doors), prove value, then expand — evolutionary, not big-bang.

Technical Architecture (MVP)

A deliberately small, simple stack. No CAD integration, no ERP rewrite — just enough infrastructure to replace the spreadsheet chaos with a structured workflow.

Layer 1: Web Application (Frontend)

The day-to-day interface for estimators

  • React + TypeScript — drawing upload, BOM editor, supplier price entry, alternatives review, proposal preview
  • Familiar table-first UX — feels like Excel, behaves like a system
  • Role-based screens — estimator, lead estimator (approver), administrator

Layer 2: Backend API

The calculation and orchestration core

  • Python (FastAPI) or Node.js — REST API for projects, BOMs, suppliers, quotes
  • Pricing rules engine — labor, finishing, urgency, volume, location, overhead, markup
  • Volume / urgency discount logic — 3–5% for large orders; configurable urgency multipliers
  • Excel + PDF generation — openpyxl + jinja2 templates for Excel; ReportLab / WeasyPrint for branded PDF

Layer 3: Data Layer

Structured storage for projects, BOMs, materials, suppliers

  • PostgreSQL — projects, BOMs, supplier catalogue, materials master, historical quotes
  • Vector index (pgvector) — for similarity search over historical projects
  • One-off cleanup pass — BA + ML engineer prepare the 70% of historical projects for ingestion

Layer 4: AI Module

The intelligence behind the assistant — kept narrow on purpose

  • LLM (GPT-4o class or local equivalent) — extract specifications from PDFs / unstructured spec sheets; reason about material substitutions
  • Embedding-based similarity search — find historical analogues for a new RFQ
  • Classical ML (XGBoost / Random Forest) — price-range estimator on top of historical data, used as a sanity check, not the source of truth

Layer 5: Integrations (Phase 2 — Optional)

Out of scope for MVP — listed for roadmap clarity

  • Supplier portals — automated price scraping or API ingestion (still manual at MVP)
  • ERP / accounting — push approved proposals into the order book
  • CRM — link quotes to opportunities for win/loss analytics

Why this is realistic

The MVP deliberately avoids the expensive parts: no CAD parsing, no live supplier APIs, no ERP integration. Those are the items that blow up scope on industrial-AI projects.
